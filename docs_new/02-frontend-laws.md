# 02 Frontend Laws

生成日期：2026-06-28

本文是 dzcom_web 前端开发铁律。若与 `dzcom_web/DEVELOPMENT_RULES.md` 或 `dzcom_web/mock` 中旧文档冲突，以本文为准。

## 最新结论区

### 最高优先级

- **本文是前端新生成代码、修改代码、页面补齐、接口接入、adapter 调整的第一优先级规则。任何前端实现必须先满足本文，再谈功能完成。**
- **不得以“先跑通”“临时演示”“后面再改”“旧页面就是这样”为理由违反铁律。铁律不满足时，必须停止前端绕过，转为更新契约、登记后端 gap 或调整方案。**
- **每次前端任务开工前必须先声明本次适用铁律；收尾时必须说明 type-check/build/目标路由冒烟情况。**

### 数据真实性

- 前端不得使用假数据、硬编码数据、临时演示数组、手填 BizId 或堆原始 JSON 绕过真实体验。
- API 未就绪时展示明确空状态、待接入状态或后端 gap，不做假图表、假列表、假工作台。
- 页面不得直接散写接口路径，必须通过 `src/entities/*/api.ts` 或共享 API client 调用。
- 页面不得在组件里临时解析复杂后端结构作为主要逻辑；DTO、ViewModel、表单模型、adapter 必须分层。

### 对象选择器

- 所有需要用户选择并回传的 `bizId`、`*BizId`、`businessBizId`、`targetBizId`、`userBizId` 必须使用对象选择器或可搜索下拉。
- 选择器必须展示可理解的 display 信息，例如名称、编码、状态、摘要、归属、最近更新时间。
- 禁止让用户手动输入业务 id 完成核心动作。
- 如果后端没有列表/搜索/分页/排序接口，必须写入 `06-backend-gap-list.md`，不能前端绕过。

### 核心模块体验

- 投资报告必须是结构化体验：摘要、指标、质量门禁、证据、风险、建议、关联对象和操作结果。
- 模拟交易必须是结构化体验：组合、持仓、订单、调仓、估值、收益、风控原因和执行反馈。
- 复盘闭环必须是结构化体验：时间线、步骤、输入输出摘要、失败原因、反馈、Prompt/模型/Skill 影响。
- 原始 JSON 只能作为辅助调试折叠区，不能成为主要业务展示。

### UI 方向

- 当前已确认 UI 方向参考阿里云控制台比例：全宽薄顶栏、约 `204px` 左侧菜单、宽广工作面、黑灰主色、紧凑工具条、标签页与表格主导。
- UI 框架详见 `24-frontend-ui-aliyun-console-reference.md`；业务页面替换必须逐页验证，不得一次性无验证大改。
- 页面视觉必须服务长期运营效率，不做营销式 hero、大面积装饰性深色背景、霓虹网格或玻璃拟态。
- 配置、审计、运行态页面优先采用左侧分组/树、主工作区、薄筛选行、标签页/表格、抽屉详情；右侧检查器只在确有追踪需求时出现。
- 列表、按钮、输入框、弹窗必须统一使用控制台比例；不得各页面自行发明高度、圆角和按钮密度。
- 站点品牌使用 DZAI 自有黑灰 icon 和黑灰主色，不使用阿里云橙、紫色高饱和主题或阿里云图标作为品牌符号。

### 错误、权限与状态

- 401、403、404、409、500 必须映射为用户可理解的状态。
- 数据不足、接口缺失、字段缺失、解析失败必须显式展示。
- 高风险动作必须有二次确认、失败原因和可追踪上下文。
- 状态、枚举、风控拒绝原因必须使用字典或后端结构化字段展示。
- 页面主展示区不得直接裸展示后端英文枚举、环境、场景、任务类型、业务类型或状态码，例如 `INVESTMENT_REPORT`、`DEFAULT`、`AUTO_INVESTMENT_REPORT_GENERATION`；必须优先用中文 label 展示，必要时把原始编码作为次要小字、Tooltip 或追踪字段保留。

### 完成标准

- 页面完成必须能通过 `npm run type-check`。
- 页面完成必须能通过 `npm run build`。
- 涉及目标路由时，必须做至少一个路由冒烟：登录、进入页面、触发核心加载、确认没有空白页或控制台级阻断。
- 如验证无法执行，必须在 handoff 写明原因、替代验证和剩余风险。
