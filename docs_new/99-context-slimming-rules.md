# 99 Context Slimming Rules

生成日期：2026-06-28

本文是后续所有 Codex 协作的硬规则。若任务执行方式与本文冲突，以本文为准。

## 最新结论区

### 默认读取规则

- 不允许每次任务都重新读取所有历史文档、所有 mock、所有前后端文件。
- 当前任务只读取：
  - `00-current-handoff.md`
  - 当前任务对应的 contract/laws/gap list 摘要
  - 当前要改的前后端文件
  - 当前验证需要的脚本或配置
- 每次开始任务前先说明本次将读取哪些文档/文件，避免上下文爆炸。
- 如果需要历史，只按明确章节读取，不全量扫。

### 文档沉淀规则

- 已完成的上下文必须沉淀到文档，后续不再靠聊天历史维持。
- 正在进行中的任务可以强关联上下文；任务完成后必须压缩成文档摘要。
- 大文档必须有“最新结论区”和“历史归档区”；默认只读最新结论区。
- 任何完成项都必须从“对话上下文”迁移到“文档上下文”。
- 后续如果发现旧文档无最新结论区，优先补最新结论区，而不是继续在对话里堆历史。

### Handoff 文件自瘦身规则

- `00-current-handoff.md` 是默认入口，不是无限流水账；它必须保持“入口索引 + 当前状态 + 下一步”定位。
- `00-current-handoff.md` 的“最新结论区”只保留当前仍会影响下一轮决策的内容：当前状态、未关闭 gap、关键契约变化、最近验证结果和下一步入口。
- 已完成且不再影响当前决策的过程项，必须迁移到“历史归档区”或独立专题文档，例如页面审计、数据审计、专项 gap、接口清单。
- 每次向 `00-current-handoff.md` 追加内容前，先判断是否应替换/合并旧摘要；禁止长期只追加不压缩。
- 当 `00-current-handoff.md` 最新结论区超过约 `150` 行，或全文超过约 `350` 行，下一次文档更新必须先瘦身：合并重复完成项、保留最近结论、把旧明细归档到专题文档。
- 默认任务只读 `00-current-handoff.md` 的最新结论区；历史归档区只在任务明确要求追溯时按章节读取。
- `00-current-handoff.md` 的历史归档每条最多保留短摘要、日期、相关文档路径和验证结论；不得塞完整 diff、完整 API 列表、完整日志或长篇过程。
- 长期统计类信息，例如“多少接口/多少页面/审计矩阵”，应沉淀到独立审计文档，handoff 只保留路径和最新结论。

### 对话控制规则

- 如果上下文开始膨胀，优先更新 handoff/gap list，而不是继续在对话里堆分析。
- 不要把完整 API 文档、完整 diff、完整日志长期塞进对话；只保留摘要、路径、关键接口、验证结果。
- 命令输出只保留关键结论；完整日志不进入长期上下文。
- 完成后必须输出：
  - 已沉淀到文档的内容。
  - 下次默认只需读取的入口。

### 文件读取策略

- 查文件用 `rg` 或 `rg --files` 优先。
- 先看目录、索引、文件名和相关摘要，再读目标文件。
- 不读构建产物、依赖目录、无关历史 mock、无关 migrations。
- 只在需要确认接口、DTO、页面或验证脚本时读取对应文件。

### 跨仓任务最小上下文

一个普通前后端联动任务默认只需要：

1. `dzcom/docs_new/00-current-handoff.md`
2. `dzcom/docs_new/01-product-contract.md` 的最新结论区
3. `dzcom/docs_new/02-frontend-laws.md` 或 `03-backend-laws.md` 的相关段落
4. `dzcom/docs_new/04-api-contract-rules.md` 的相关段落
5. `dzcom/docs_new/06-backend-gap-list.md` 中相关 gap
6. 本次涉及的 Controller/DTO/Service/migration/test
7. 本次涉及的 frontend api/model/adapter/page/router
8. 本次验证命令需要的 `pom.xml` 或 `package.json`

### 2026-06-29 Handoff 自瘦身

补充 `00-current-handoff.md` 自身瘦身硬规则：handoff 只做默认入口和当前状态索引，超过阈值必须先压缩，再追加新结论。
