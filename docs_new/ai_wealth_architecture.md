## 1 目标与范围

- 目标用户：有自主投资需求的零售用户、高净值个人、以及机构（可选）。
- 目标能力：
  - 聚合多源市场数据（行情、公告、研报、宏观数据等）。
  - AI 驱动的多维分析（因子分析、情绪分析、组合优化、风险预测、择时建议）。
  - 智能可视化（交互式图表、趋势/回测/模拟）。
  - 产品筛选/策略推荐与可执行的一键投资/撤出（通过第三方券商/托管/支付通道）。
  - 完整的权限、合规、审计与风控链路。

---

## 2 主要业务域（Domain）

将系统按业务域划分，便于后续微服务或模块化实现：

1. 用户域（User）
   - 用户身份、认证（含 KYC）、权限、首选项、风险承受能力。
2. 市场数据域（Market Data）
   - 实时行情、历史时序、新闻、公告、研报、宏观指标、链上数据（若支持数字资产）。
3. 产品域（Product Catalog）
   - 基金、股票、期货、期权、ETF、债券、结构性产品、第三方理财产品的统一元数据与合规信息库。
4. 组合与投资域（Portfolio & Orders）
   - 用户持仓、组合估值、订单管理、执行记录、对账与结算。
5. AI/分析域（AI Engine）
   - 数据预处理、特征仓库、模型推理服务（因子打分、资产配置、信号生成、情绪与事件分析、回测模拟）。
6. 风控与合规模块（Risk & Compliance）
   - 风险限额、风控规则、异常检测、合规检查（交易前/交易后）、AML 规则。
7. 支付与结算域（Payments & Custody）
   - 支付通道、资金划转、托管/托管人对接、第三方券商下单接口。
8. 通知与交互域（Notifications、UI）
   - 告警、消息、推送、仪表盘、图表引擎、交互式策略编辑和可视化。
9. 运维与监控（Observability）
   - 日志、指标、链路追踪、审计日志、SLA 监控。

## 3 模块化后端架构建议

建议采用模块化微服务或模块化单体架构，初期可从模块化单体（易于交付）起步，随着业务扩展逐步拆分为微服务。核心服务/模块（按职责）：

- API Gateway / Edge
  - 统一鉴权、流量控制、路由、协议转换（REST / WebSocket / gRPC）。
- Auth & User Service
  - 登录（OAuth2 / JWT）、KYC、2FA、权限管理、用户偏好与配置。
- Market Data Ingest & Storage
  - 实时行情采集（接入多家数据源）、历史数据 ETL、时序数据库（InfluxDB、TimescaleDB 或 ClickHouse）用于高并发查询。
- Product Catalog Service
  - 统一管理所有理财产品元数据、合规信息、费率、可交易渠道、最小/最大份额等。
- Portfolio & Position Service
  - 持仓管理、估值引擎、收益率计算、盈亏统计。
- Order Execution Service
  - 下单逻辑、订单路由、确认/回执、失败重试、保证幂等性；与券商 / 托管 / 支付网关对接。
- AI Analysis Service（Model Serving）
  - 独立部署的模型推理层（可用 KFServing、Seldon 或自建 gRPC 推理服务），提供因子评分、信号、组合建议、情绪分析 API。必须包含模型元数据、版本与审计。
- Feature Store & Data Pipeline
  - 用于模型训练和实时推理的特征平台（可用 Feast / 自研），支持离线/在线特征。
- Backtesting & Simulation Service
  - 策略回测、蒙特卡洛模拟、情景测试与绩效报告。
- Risk & Compliance Service
  - 实时风控规则、交易前校验、异常检测、AML 报表、审计日志存储。
- Notification Service
  - 邮件、短信、App Push、站内信与交易流水通知。
- Scheduler / Job Worker
  - 定时任务（净值计算、再平衡提醒、数据清洗、模型训练调度）。
- Common Infrastructure
  - 配置中心、服务注册与发现、日志聚合（ELK/EFK）、指标（Prometheus）、Trace（Jaeger）、Secrets 管理。

消息中间件：Kafka / RabbitMQ 用于事件驱动（市场数据流、订单事件、风险告警、模型生成事件等）。

存储建议：

- 关系数据库（PostgreSQL / MySQL）用于事务、用户与订单数据（强一致性）；支持主从与备份。
- 时序与分析数据库（ClickHouse/TimescaleDB/InfluxDB）用于行情、指标与高并发 OLAP 查询。
- 对象存储（S3）用于大文件、模型权重、历史回测快照。

## 4 前端架构建议

前端将面向 Web/移动端（响应式或独立移动应用）。建议采用现代 SPA 框架：React（TypeScript）或 Vue 3（TypeScript）。

关键模块：

- 登录与用户中心（包括 KYC 引导）
- 仪表盘（Portfolio Overview）：净值、今日盈亏、资产配置占比、风险指标
- 市场与产品页：多资产筛选、产品详情、费率与合规信息
- AI 分析页：AI 推荐、因子打分、选股/选基策略、情绪与事件分析
- 交互式图表：K 线、成交量、技术指标、回测曲线（建议用 Highcharts、ECharts、TradingView 或类似图表库）
- 策略编辑与回测界面：可视化策略搭建、回测参数输入、结果展示
- 订单与资金页：一键投资/撤出、订单状态、历史记录
- 通知中心：消息、告警

实时性：

- 行情与订单实时状态使用 WebSocket 或双向 gRPC（Web）推送。
- 对于图表大数据量，后端需提供分页/下采样 API（downsampling/aggregation）。

安全与 UX：

- 在发起投资/撤出等敏感操作时，要求二次确认（2FA / 短信验证码 / 指纹）和显示风险提示。

## 5 关键数据流与交互序列（示例）

场景 A — 用户一键投资流程（高层次）

1. 用户在前端点击“一键投资”并选择产品与金额。
2. 前端调用 API Gateway -> Portfolio Service 验证用户身份与额度。
3. Portfolio Service 调用 Risk Service 做交易前风控校验。
4. 风控通过后，Order Service 生成订单并写入事务数据库（Pending 状态），事件发送到消息总线（order.created）。
5. Order Execution Service 订阅事件并路由到对应券商/托管 API 执行下单。
6. 券商回执到 Order Execution Service -> 更新订单状态 -> 发送 order.executed 事件。
7. Portfolio Service 接收回执，更新持仓与估值；通知 Notification Service 推送结果。
8. Audit Service 记录完整交易链路与审计日志。

场景 B — AI 推荐流程（异步）

1. Data Ingest 将原始数据写入时序/对象存储并触发 ETL 作业。
2. Feature Store 生成特征并供离线训练或在线推理使用。
3. AI Engine 定时或触发式运行模型，生成推荐信号并写入模型输出表 / 事件总线（signal.generated）。
4. 前端通过 API 查询最新信号或订阅信号推送，展示给用户并允许用户转为策略/下单。

## 6 非功能需求（NFR）

- 可用性：核心交易链路 99.95%+（多可用区部署、自动故障恢复）。
- 延迟：行情展示 <200ms（cache、CDN）；下单响应尽量<1s（取决于券商）。
- 吞吐量：支持高并发行情订阅（数万/秒）与订单并发（依规模扩展）。
- 安全：全链路加密（TLS）、敏感信息加密存储、基于角色的访问控制（RBAC）。
- 可扩展性：各服务水平扩展、事件驱动架构保证松耦合。
- 可观测性：端到端链路追踪、业务与指标报警、日志与审计存档。

## 7 合规、审计与风控要点

- KYC / AML：接入第三方 KYC 服务，交易行为异常检测、可疑交易上报链路。
- 审计日志：所有交易和关键操作写入不可篡改审计库（写后只追加），并支持导出与监管查询。
- 资金托管：若涉及代为交易或代管资金，必须与合规的托管平台/银行合作并保存对账记录。
- 合规规则引擎：交易前合规校验（产品是否允许该类客户投资、额度限制、黑名单等）。

## 8 ML 与 AI 实施细则（治理方向）

- 模型治理：模型版本、指标（AUC/Sharpe/回撤）、训练数据快照、注册表（Model Registry）。
- 可解释性：关键模型需输出可解释文件（特征重要性、置信度），供风控与合规审查。
- 幻觉/错误控制：AI 生成的信号/说明仅作为建议；任何涉及资金变动的自动化动作必须有人复核或硬性规则触发。
- 数据质量：数据溯源与校验（时间对齐、缺失填充、异常值处理）。
- 自动/手动回退：上线新模型需 A/B 或金丝雀策略，实时监控信号质量指标，必要时快速回退。

## 9 技术选型建议（参考）

- 后端框架：Spring Boot 4（Java / Kotlin）
- API：REST + gRPC（内部服务）
- 数据库：Postgres / MySQL（事务）；ClickHouse / TimescaleDB（分析/时序）
- 缓存：Redis
- 消息：Kafka（高吞吐）
- 模型部署：Seldon / KFServing / TorchServe 或自研 gRPC 推理服务
- 部署：Kubernetes（多可用区）、Helm
- CI/CD：GitHub Actions / Jenkins
- 监控：Prometheus + Grafana，日志 ELK/EFK，链路 Jaeger
- 图表：ECharts / TradingView / Highcharts

## 10 MVP 与分阶段实现建议

阶段 0（准备）
- 项目脚手架、用户认证与基础 DB。
- 简单市场数据接入（股票/基金实时与历史）和产品目录。
- 基础前端（登录、仪表盘、产品列表）。

阶段 1（核心交易能力 + AI 最小集）
- 一键投资/撤出集成一个或多个券商沙箱接口（订单下发、回执、对账）。
- 简单 AI 推荐：基于规则+轻量 ML 的因子打分与推荐。
- Portfolio 显示与基本图表（净值、收益曲线）。

阶段 2（增强 AI 与风控）
- 增加因子组合、模型回测、在线特征与 Feature Store。
- 风控引擎、合规模块、审计链路完善。

阶段 3（规模化与多资产）
- 接入更多资产（期货、期权、债券、数字资产）。
- 模型治理、A/B、自动化交易（在严格风控与合规下逐步开放）。

## 11 风险与注意事项

- 数据延迟/不一致可能导致错误信号，需做好监控与降级策略（例如行情延迟时禁用自动下单）。
- AI 产生错误或误导性建议（幻觉）会带来合规与法律风险，强制人工复核与可追溯说明。
- 第三方依赖风险（券商、数据商）需要冗余供应商和降级方案。
